Hazard Warning in Manhattan Grid

Aim of the example

The example models a vehicular communication network for hazard warning in urban environment with Manhattan grid topology. The aim is to measure the performance of V2V communication in avoiding the hazard under various conditions.

MATLAB-NS3 Interface

The core simulation runs in NS-3 with scenario description written in MATLAB. The NS-3 library functions are called via Mex-bindings interface from MATLAB. Along with the scenario, the features implemented in MATLAB are:
  • Manhattan-grid topology creation.
  • Routing of vehicles.
  • Re-routing of vehicles.
  • Packet construction of all types.
  • Mobility intelligence based on the received packets.
  • Post-simulation visualization.

Scenario Description

The scenario consists of vehicles moving in a Manhattan grid of roads. It includes:
  • Several ‘regular’ vehicles (shown as numbers) that need to avoid the hazard to reach the destination
  • A broke down ‘hazard’ vehicle (red cross mark) that pops up on one of the roads
  • A set of ‘rogue’ vehicles (small black boxes) that cause communication interference
  • Each vehicle is installed with WAVE (over 802.11p) stack for V2V communication.
Following figure has a 4x4 grid.
Each road in the grid can be identified by a 3-tuple – [direction, block horizontal index, block vertical index]. The last 2 tuple members identify a block and ‘direction’ specifies the road surrounding that block. There can be 4 roads surrounding a block in +x, -x, +y, and -y directions as shown below.
Block’s horizontal index varies from 1 to total horizontal blocks. Likewise, vertical index varies from 1 to total vertical blocks. So, a sample valid road identifier could be [‘+y’, 2, 3] which is +y directional road of the 2nd block in the 3rd row (from bottom) of the grid.
Note: As shown in visualizer figure, there are no roads at the boundaries of grid. So, a block may have less than 4 roads surrounding it. For example: 4th block in first (or any) row of grid does not have ‘-x’ and ‘+y’ roads. So [‘-x’, 4, 1] is not a valid road.

The routing algorithm finds a route from start of source road to end of destination road, during the simulation setup. To make the scenario interesting, the vehicle route derivation logic ensures that the route passes through the road with hazard based on configured hazard location defined by ‘hazardLoc’ parameter in the MATLAB scenario file. ‘hazardLoc’ is again a valid road identifier defined using above described 3-tuple identification.
The rogue vehicles, on the other hand just hover around the hazard with the sole purpose of creating network interference around it. User just need to configure the rogue vehicle count, and their route is automatically derived to make sure they move in vicinity of hazard.
A hazard appears on one of the roads (configured using ‘hazardLoc’ parameter) at a pre-configured time during the simulation run. It periodically broadcasts a ‘hazard warning’ packet with its location in the form of blocked road ID.
The regular vehicles react to the warning message differently, based on their location.
  • If the vehicle is already on the road with hazard before receiving the hazard warning, it just stops to avoid collision with the broken-down vehicle.
  • If the vehicle is next headed towards hazard road, it dynamically finds an alternate route (if possible) and thus avoids the hazard.
  • If the vehicle is not headed towards hazard, it does not react and continue along its journey.

What is not in scope of this demo

  • The only mobility intelligence in this example is to avoid the hazard. There is no collision avoidance mechanism among other vehicles so there is possibility of them running over each other.
  • The movement over intersection area of the roads is not modeled. The vehicles, after reaching the end of one road go to the start of next road in their journey in zero time.

Communication model

In this scenario, the following packet types are sent periodically:
  • ‘Hazard Warning’ Packet – Transmitted by broken down vehicle with information of position of hazard (as road ID).
  • ‘Position beacon’ Packet – Transmitted by both regular vehicles and rogue vehicles.
It is quite possible that some regular vehicles may not receive the warning message in time to either re-route or avoid the hazard for the following reasons:
  • Channel busy with data traffic from other vehicles
  • Interference
  • Path loss
  • Low periodicity of warning messages
NOTE: All the packets are sent and received on CCH channel as WSMP (WAVE Short Message Protocol) packets. The payload of the WSMP message is specific to this example.

Results and Analysis

While the simulation is running, the results are gathered into log files. The results include:
a) Packet statistics at application level for each regular vehicle and broken-down(hazard) vehicle.
  • Number of ‘hazard warning’ transmitted. (0 for regular vehicle).
  • Number of ‘position beacons’ transmitted. (0 for hazard).
  • Number of ‘hazard warning’ received. (0 for hazard).
  • Total number of packets received successfully including those from rogue vehicles.
b) MAC/PHY statistics for each regular vehicle and broken-down(hazard) vehicle.
  • Total number of corrupted Rx packets.
c) Success/Failure counts in terms of number of vehicles able to avoid taking the road with hazard. Also, a count of hard failure is shown which is updated when a vehicle not only takes the route with hazard but collides with it.
A visualization script reads the simulation results from log files, analyzes them and performs visualization. The visualization includes:
  • Graphical visualization of Manhattan-grid and movement of all vehicles.
  • Hazard is shown as red cross (X).
  • Vehicles which could not avoid the hazard and stopped on the road with hazard are shown in red numbers. On the other hand, a stopped vehicle which is not in red signifies that it has completed its journey.
  • Vehicles which collide with hazard are shown as black crosses (X).
  • An event-log is shown, which is text-commentary of major events as they happen during the visualization such as: Hazard avoidance by taking alternate route, Hazard collision, stopping on the hazard road to avoid collision.
  • A slider to vary the visualization speed, and a button to pause/play the visualization.
